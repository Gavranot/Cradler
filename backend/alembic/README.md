# Database Migrations

This directory contains Alembic database migrations for the Cradler application.

## Quick Start

### Apply all pending migrations

```bash
# From backend directory
python migrate.py upgrade

# Or using alembic directly
alembic upgrade head

# Using Docker
docker-compose exec backend alembic upgrade head
```

### Create a new migration

```bash
# From backend directory
python migrate.py revision "add user profile fields"

# Or using alembic directly
alembic revision --autogenerate -m "add user profile fields"

# Using Docker
docker-compose exec backend alembic revision --autogenerate -m "add user profile fields"
```

### Rollback last migration

```bash
# From backend directory
python migrate.py downgrade

# Or using alembic directly
alembic downgrade -1

# Using Docker
docker-compose exec backend alembic downgrade -1
```

### View migration history

```bash
# From backend directory
python migrate.py history

# Or using alembic directly
alembic history --verbose

# Using Docker
docker-compose exec backend alembic history --verbose
```

### Check current database version

```bash
# From backend directory
python migrate.py current

# Or using alembic directly
alembic current

# Using Docker
docker-compose exec backend alembic current
```

## Migration Files

Migrations are stored in `alembic/versions/` directory. Each migration file contains:
- `upgrade()` function: Applied when migrating forward
- `downgrade()` function: Applied when rolling back

### Current Migrations

1. **001_initial_schema.py** - Creates all initial tables:
   - `users` - User accounts and authentication
   - `scrapers` - Scraper configurations
   - `scraping_runs` - Execution history
   - `scraper_templates` - Platform-specific templates
   - `scraping_knowledge` - Vector embeddings for RAG (pgvector)
   - `chat_sessions` - Chat conversation history

## Important Notes

### pgvector Extension

The initial migration creates the `vector` PostgreSQL extension required for the `scraping_knowledge` table. This extension must be available in your PostgreSQL installation.

If you're using the Docker setup, the `pgvector/pgvector:pg17` image includes this extension.

### Async Support

Our migrations are configured to work with async SQLAlchemy. The `env.py` file handles async database connections properly.

### Autogenerate Limitations

When using `--autogenerate`, Alembic may not detect:
- Table or column renames (appears as drop + create)
- Changes to constraints in some cases
- Changes to custom types or indexes

Always review generated migrations before applying them.

## Common Tasks

### Reset database (DESTRUCTIVE)

```bash
# Downgrade all migrations
alembic downgrade base

# Then upgrade back to latest
alembic upgrade head
```

### Skip a migration (mark as applied without running)

```bash
# Stamp the database at a specific revision
alembic stamp <revision_id>

# Example: Mark initial schema as applied
alembic stamp 001
```

### Create empty migration (manual changes)

```bash
alembic revision -m "add custom index"
# Then edit the generated file to add your custom changes
```

## Troubleshooting

### "Can't locate revision identified by 'X'"

This usually means your database is out of sync with migrations. Check current version:

```bash
alembic current
```

### "relation already exists"

The table already exists in your database. You can either:
1. Drop the table manually or reset the database
2. Stamp the database to skip that migration: `alembic stamp <revision>`

### Import errors in env.py

Make sure you're running from the `backend` directory, and all dependencies are installed:

```bash
cd backend
pip install -r requirements.txt
```

## Best Practices

1. **Always review autogenerated migrations** before applying them
2. **Test migrations on a copy of production data** before deploying
3. **Never modify applied migrations** - create a new one instead
4. **Keep migrations small and focused** - one logical change per migration
5. **Add data migrations carefully** - use batching for large datasets
6. **Include both upgrade and downgrade** - always test rollback path

## Environment Configuration

Migrations read the database URL from the `DATABASE_URL` environment variable (via `core.config.settings`).

Make sure your `.env` file is properly configured:

```env
DATABASE_URL=postgresql://cradler:password@localhost:5432/cradler
```
